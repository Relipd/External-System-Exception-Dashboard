## 〇、需求背景与演进

### 原始需求

- **主表（异常明细表）**：`部门 — 平台 — 系统编码 — 异常编码 — 风险等级 — 数量`
- **附表（时间趋势表）**：`时间 — 部门 — 系统编码 — 待处理/审阅/超时`

两张表通过 **部门 + 系统编码** 可以关联。目标是通过趋势表的时间窗口，过滤主表数据，实现 **时间 → 系统编码 → 平台 → 部门** 的穿透查询，让所有看板组件都响应时间筛选。

### 需求演进

| 阶段 | 需求 | 实现 |
|------|------|------|
| 1 | 附表添加「权限中心-系统编码」字段，并能按时间展开 | `ITrendFieldMap` + `ITrendRecord` 新增 `systemCodeFieldId` / `systemCode` |
| 2 | 趋势图支持按系统编码维度查看 | `computeTrendChartData` 增加 `groupKey` 参数 |
| 3 | 取消系统编码维度切换，趋势图始终按部门展示 | 移除 `trendGroupKey` 状态和切换按钮 |
| 4 | 已超时卡片显示为零 | `_totalOverdue` 独立保存，不受截断影响 |
| 5 | 趋势数据保存超限（customConfig 4096） | 紧凑格式存储、智能截断保留最新记录 |
| 6 | **时间维度贯穿全表**（当前状态） | `timeActivePairs` + 统一时间过滤链 |
| 7 | 趋势图的时间逻辑统一作用于所有表单 | 过滤逻辑提到 App.tsx 层，TrendWarning 不再内部过滤 |## 一、总览

时间维度（7天/1月/3月/6月/1年）是仪表盘的核心筛选维度。所有看板组件——风险总览卡片、异常分布图、系统排行、趋势折线图——都基于**同一套时间窗口**计算数据。

**关键原则**：时间过滤只在一个地方做，结果供全表共享。

### 数据流总图

```
飞书趋势表（全量 60 条）
  │ saveConfig() 时读取（Config 模式有权限）
  ▼
customConfig._trendData（紧凑格式，截断后约 28 条）
  │ loadConfig() 时恢复
  ▼
trendRecords（App.tsx 状态）
  │
  ├──▶ timeFilteredTrends ──▶ timeActivePairs ──▶ filteredDetails ──▶ analytics（所有图表）
  │         (按 timeRange 过滤)   (部门|系统编码 对)   (时间 + 部门过滤)
  │
  └──▶ timeFilteredTrends ──▶ filteredTrendRecords ──▶ TrendWarning
                                    (叠加部门过滤)         (不再内部过滤)
```

## 二、时间范围定义

### 2.1 选项列表

```typescript
// App.tsx — 按钮渲染用
const TIME_RANGES = [
  { key: '7d',  label: '近七天' },
  { key: '1m',  label: '近一个月' },
  { key: '3m',  label: '近一个季度' },
  { key: '6m',  label: '近半年' },
  { key: '1y',  label: '近一年' },
];

// analytics.ts — 天数映射（计算用）
export const RANGE_DAYS = {
  '7d': 7, '1m': 30, '3m': 90, '6m': 180, '1y': 365,
};
```

### 2.2 截止日期计算

```typescript
// analytics.ts
export function getCutoffDateStr(days: number): string {
  // 1. 获取当前本地日期（Intl 保证不受浏览器时区影响）
  const tz = Intl.DateTimeFormat().resolvedOptions().timeZone;
  const fmt = new Intl.DateTimeFormat('en-CA', { timeZone: tz, ... });
  const todayStr = fmt.format(new Date());  // → "2026-06-17"
  
  // 2. 构造 UTC 日期，往前推 N 天
  const today = new Date(todayStr + 'T00:00:00Z');
  const cutoff = new Date(today);
  cutoff.setUTCDate(cutoff.getUTCDate() - days);
  
  // 3. 输出 yyyy-MM-dd
  return dateToStr(cutoff);
}
```

- 始终使用**当地时区**（Intl.DateTimeFormat resolvedOptions）
- 始终用 **UTC 日期运算**避免夏令时偏移
- 截止日期 = 当天往前推 N 天，当天不计入

## 三、核心过滤逻辑（App.tsx）

### 3.1 第一步：时间过滤趋势数据

```typescript
const timeFilteredTrends = useMemo(() => {
  if (trendRecords.length === 0) return trendRecords;
  const days = RANGE_DAYS[timeRange] || 90;
  const cutoffStr = getCutoffDateStr(days);
  return trendRecords.filter((r) => r.date >= cutoffStr);
}, [trendRecords, timeRange]);
```

结果：`timeFilteredTrends` = 仅包含时间窗口内的趋势记录

### 3.2 第二步：提取活跃（部门, 系统编码）对

```typescript
const timeActivePairs = useMemo(() => {
  const pairs = new Set<string>();
  for (const r of timeFilteredTrends) {
    if (r.department && r.systemCode) {
      pairs.add(`${r.department}|${r.systemCode}`);
    }
  }
  return pairs.size > 0 ? pairs : null;
}, [timeFilteredTrends]);
```

结果：`timeActivePairs` = 时间窗口内出现过异常的 `部门|系统编码` 集合

### 3.3 第三步：过滤明细数据

```typescript
const filteredDetails = useMemo(() => {
  let result = details;
  // 按时间活跃对过滤
  if (timeActivePairs?.size > 0) {
    result = result.filter((d) => timeActivePairs.has(`${d.department}|${d.systemCode}`));
  }
  // 按部门过滤（在时间过滤之上叠加）
  if (globalDepartments.size > 0 && allDepartments.length > 0) {
    result = result.filter((d) => globalDepartments.has(d.department));
  }
  return result;
}, [details, timeActivePairs, globalDepartments, allDepartments]);
```

**过滤顺序**：先时间 → 再部门

### 3.4 第四步：过滤趋势数据（给 TrendWarning）

```typescript
const filteredTrendRecords = useMemo(() => {
  let result = timeFilteredTrends;       // 已按时间过滤
  if (globalDepartments.size > 0 && ...) {
    result = result.filter((r) => globalDepartments.has(r.department));
  }
  return result;
}, [timeFilteredTrends, globalDepartments, allDepartments]);
```

**关键**：`filteredTrendRecords` 传给 TrendWarning 后，TrendWarning **不再做任何时间过滤**——直接使用。

### 3.5 口径切换

```typescript
const analytics = computeAnalytics(filteredDetails, countMode);
// countMode: 'pending' | 'total' | 'processed'
```

口径（待处理/完成核查/历史总计）在时间过滤之后应用，选择使用哪个 count 字段。

## 四、趋势图图表构建（TrendWarning）

TrendWarning 自身**不再有**时间过滤逻辑，只负责图表渲染。

### 4.1 聚合粒度自适应

```typescript
// TrendWarning 内部 chartData useMemo
if (timeRange === '7d' || timeRange === '1m')  → 按日显示 + 补齐缺失日
if (timeRange === '3m')                         → 按周聚合 + 补齐缺失周
if (timeRange === '6m' || timeRange === '1y')   → 按月聚合 + 补齐缺失月
```

- 短时间范围（7天/1月）：每日一个数据点，缺失日填 0
- 中等范围（3月）：按周聚合，缺失周填 0
- 大范围（6月/1年）：按月聚合，缺失月填 0

### 4.2 有数据的日期补齐

```typescript
function padDateRange(data, groups, stepDays)     // 填补缺失日期
function padToFullRange(data, groups, start, end, stepDays) // 强制覆盖完整范围
function padMonthly(data, groups, rangeStart, rangeEnd) // 按月补齐
function aggregateChartData(data, groups, 'week'|'month') // 按周/月聚合
```

保证折线图连续不断裂。

## 五、保存时的时间数据流

### 5.1 数据保存链路

```
Config 模式
  读取趋势表（loadTrendRecords，API 有权限）
    → 全量 60 条
    → 预计算 _totalOverdue = SUM(overdueCount)
    → 截断为紧凑格式（字段名缩写：d,dept,code,r,p,o）
    → 保存到 customConfig._trendData（上限 4096 字符）

View 模式
  读取 customConfig
    → 恢复 _trendData（兼容新旧格式）
    → 恢复 _totalOverdue
```

### 5.2 紧凑存储格式

```typescript
// 保存时：字段名缩写，节省约 36% 空间
type TrendCompact = {
  d: string;     // date
  dept: string;  // department
  code: string;  // systemCode
  r: number;     // reviewCount
  p: number;     // pendingCount
  o: number;     // overdueCount
};

// 加载时：自动转换回 ITrendRecord
const record = {
  date: raw.date || raw.d,
  department: raw.department || raw.dept,
  systemCode: raw.systemCode || raw.code,
  reviewCount: raw.reviewCount ?? raw.r,

```

## 七、注意事项

1. **View 模式不可读取趋势表**：Dashboard 插件在 View 模式下不能访问非 `dataConditions` 的表。趋势数据**必须**在 Config 模式保存到 `customConfig`。
2. **customConfig 4096 字符上限**：紧凑格式可存约 28 条记录。超出时自动截断（保留最新记录）。
3. **时区一致性**：所有日期计算使用 `Intl.DateTimeFormat` + UTC，确保不同时区用户看到一致的时间窗口。
4. **部门筛选在时间筛选之后**：先算时间窗口内的活跃对，再按部门进一步缩小范围。
